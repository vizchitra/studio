# Implementation Design Decisions

`architecture/` defines what Studio is and why. This document is the
narrower "how we're actually building it" layer — concrete choices made
during bootstrap that aren't spelled out in the architecture docs. Treat
it as a lightweight decision log, not a spec; add to it as you make
similarly consequential calls, don't rewrite history in it.

## Monorepo, npm workspaces

`apps/`, `packages/`, `services/` per `Studio Implementation Guide.md`.
Chose plain npm workspaces over pnpm/turborepo for now — fewer moving
parts for a small team, revisit if build times become a problem once
`services/publishing` and `services/search` are real.

## IDs are ULIDs, not UUIDv4

Sortable by creation time, which makes D1 range queries and manual
debugging (e.g. "what came in around this timestamp") much easier than
random UUIDs. Generated by `packages/shared`'s monotonic factory so IDs
created in the same millisecond still sort correctly.

## D1 schema: one table per core entity, not a single polymorphic table

`Studio Data Model.md` describes a shared Identity shape (id, status,
timestamps, audit fields) across all entities. Could have modeled this as
one `entity` table with a `type` discriminator and a JSON payload column.
Chose per-entity tables instead — D1 is SQLite, and real columns with
real types/indexes/FK constraints beat a JSON blob for anything we'll
actually query (sessions by event, content by author, etc). The shared
shape is enforced by convention (every migration repeats the same
Identity columns) and by the TypeScript types in `packages/domain`, not
by the database.

Cross-cutting concerns that attach to any entity (Activity, Comment,
Note, Task, Notification, Permission, SearchIndex) *are* polymorphic —
`entity_id` + `entity_type` — because duplicating a Comments table per
entity type would be worse.

## Asset has no owning foreign key

`Media Architecture.md` says Assets are reusable across Studio and
referenced from many places. Rather than putting `event_id`,
`session_id`, `person_id` etc. as nullable columns on `asset` (most of
which would be null most of the time, and which don't capture
many-to-many cases like one photo used in two articles), all
asset-to-entity associations go through the generic `relationship`
table. The tradeoff: looking up "all assets for this session" is a join
instead of a column scan. Acceptable at current scale; revisit with a
denormalized lookup table if it becomes a hot path.

## Pipeline steps are retryable via `asset_pipeline_run`, chained via Queues

Each processing step writes a row (`pending → running → done/failed`)
before/after running, and enqueues the *next* step itself rather than a
central orchestrator dispatching all steps. This keeps `services/media`
simple (one handler, one table) and makes "what stage is this asset
stuck at" a single query. Failure just means the message isn't acked —
Cloudflare Queues redelivers per the `max_retries` in
`services/media/wrangler.toml`.

## Auth: Cloudflare Access + Google as the login provider, zero app code

Decided over building Google OAuth directly into SvelteKit. Access
handles perimeter (who reaches the Worker at all) before a request hits
application code; the app just reads the `Cf-Access-Authenticated-User-Email`
header. Tradeoff: the app trusts a header for UX-level identity but
should verify the `Cf-Access-Jwt-Assertion` JWT before any write or
permission check — see the comment in `apps/studio/src/hooks.server.ts`.
Not yet implemented; do this before the app handles anything beyond
read-only pages.

## Deploy target: Workers (with Assets), not Pages

`Studio Implementation Guide.md` lists "Cloudflare Workers" explicitly,
not Pages. `apps/studio` uses `@sveltejs/adapter-cloudflare` targeting a
Worker with a static-assets binding rather than Cloudflare Pages, so
`apps/studio` and `services/media` deploy the same way (`wrangler
deploy`) and share the same D1/R2/Queue binding conventions.

## Dependency versions were stale, and why

The first scaffolding pass pinned `svelte@4`, `@sveltejs/kit@2.0`,
`vite@5.1`, `wrangler@3.50`, TypeScript `5.4`, etc. Those numbers came
from the agent's training data, which has a real cutoff — they were
roughly a year out of date the moment they were written, not a
deliberate choice. There was no build or install run against them at
scaffold time to catch that.

Fixed by web-searching current versions and then actually running
`npm install` against the resulting dependency set in a scratch
directory before touching the repo, to catch resolution conflicts before
they become someone else's problem. That surfaced a real one: SvelteKit
2.69's peer dependency caps TypeScript at `^6.0.0` — it doesn't support
TypeScript 7 (the Go-ported "tsgo" rewrite) yet. So the monorepo is on
TypeScript 6, not 7, even though 7 is current — revisit once SvelteKit's
peer range moves.

General rule going forward: don't hand-write version numbers into
`package.json` from memory. Either check the registry or run the install
and let it fail loudly before committing.

## Linting and formatting: oxlint + oxfmt, not ESLint + Prettier

Chose the Rust-based oxc toolchain over ESLint/Prettier: one dependency
tree instead of two, no config plugins to keep in sync with the Svelte
version, and it's fast enough that `npm run lint` / `format:check` add
negligible time to CI. `oxfmt` reached Prettier-compatible output
(passing Prettier's own conformance suite) before this was adopted, so
the risk of it doing something surprising to the codebase is low.

Deliberately scoped `.oxfmtrc.json` to exclude `**/*.md` — the
architecture docs use a pandoc-derived house style (`-   ` bullets,
`------` horizontal rules) that a generic markdown formatter would
flatten to plain `- ` bullets. That's a formatting-convention difference,
not a bug, so it's left alone rather than "fixed."

`.oxlintrc.json` and `.oxfmtrc.json` at the repo root apply to everything
under it — no per-package config needed.

## Face detection (bounding boxes): Moondream 3.1 on Workers AI, not self-hosted ONNX

`face_clustering` needs face bounding boxes before anything else in the
face pipeline can happen. Considered Moondream 3.1 (a VLM, called via the
Workers AI binding) against a self-hosted specialist model — e.g.
MediaPipe's BlazeFace, an ONNX model bundled into the Worker and run
in-process via a WASM runtime, the same pattern `@cf-wasm/photon` already
uses for preview_generation.

At this project's volume (thousands, not millions, of images) Workers
AI's per-inference cost is negligible either way, so cost didn't decide
it. What did: Moondream is one integration that also covers `ocr` and
`vision_tagging` — both still stubs that will need a vision model of
their own regardless. BlazeFace only solves face detection, so choosing
it would still leave two more model decisions to make later. Moondream's
extra network hop to the Workers AI service doesn't matter here because
this is an async batch pipeline, not a user-facing request — nothing is
waiting on it in real time. Bundling an ONNX runtime and model into the
Worker also risks the bundle-size/CPU-time ceilings Workers AI sidesteps
entirely.

Revisit only if `ocr`/`vision_tagging` end up needing a different model
than Moondream anyway, which would remove the reuse argument.

## Photo attribution: `captured_by` is separate from `created_by`, never inferred from the uploader

Every entity already has `created_by`/`updated_by` — audit fields for who
performed the action in the system. Attribution ("who took this photo")
is a different fact, and conflating them breaks as soon as one person
uploads photos they didn't shoot — a volunteer coordinator collecting
from several photographers, or an admin running a bulk import of an
official photographer's work. So `captured_by` is a `Relationship`
(Asset → Person or Asset → Organisation), resolved independently per
import path, and required before publish (`architecture/Studio Data
Model.md`'s publish check refuses an Asset with none).

Resolution differs by path, not by one shared default:

- **Zip bulk import** — the filename's code token is looked up in
  `services/media/photographer-codes.json` against a specific Person.
  Unresolved codes (or Historical Import batches with no code at all)
  fall back to the VizChitra Organisation, resolved via
  `getVizchitraOrganisationId` (get-or-create at first use, not a
  migration seed row — no fixture data to keep in sync).
- **Direct upload** (volunteer/attendee, one or few files) — a required
  prompt at upload time, defaulting the suggested name to the uploader
  but always changeable. Never silently assumed from who's logged in.

The filename parsing pattern itself is a configurable per-batch template
(`import_batch.filename_template`), not a hardcoded regex — photographer
naming conventions differ by person and by year, and hardcoding one
would need a code change every time a convention shifted.

## Open questions (not yet decided)

- R2 key naming convention beyond `originals/{assetId}/{filename}` — not
  a current problem (each upload gets a fresh ULID `assetId`, so no
  collisions yet), but worth a real convention before Historical/Mixed
  import modes land (`architecture/Media Architecture.md`, Import Modes).
- Whether `search_index` stays a plain D1 table or moves to D1 FTS5 /
  an external index once query patterns from real usage are known.
- Where the face *embedding* for `reference_person_matching` comes from
  (`face_clustering`'s Moondream call only produces bounding boxes, not a
  vector) — a separate decision from detection, still open.
